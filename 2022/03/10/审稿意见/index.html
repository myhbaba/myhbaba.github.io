SUBMISSION: 47
TITLE: A Fault-Tolerant Scheduling Algorithm with Replication Strategy of Minimum Impact on Real-Time Performance for Real-Time Heterogeneous Systems
AUTHORS: Dengfeng Mao, Wei Hu, Jing Liu and Yu Gan

———– Novelty / Originality ———–
SCORE: 3 (Fair (This work extends existing work, but provides significant new contributions))
———– Quality of the Presentation ———–
SCORE: 4 (Good (The presentation is very clear and easy to follow))
———– Technical Strength ———–
SCORE: 3 (Fair (The work in this paper is correct, but its validation methodology is limited or flawed))
———– Significance of Work ———–
SCORE: 3 (Fair (Attendees will find this work interesting and may build upon it))
———– Relevance to the Conference ———–
SCORE: 5 (Excellent (This is exactly the type of topic I want to see be presented at CCGrid))
———– Reviewer’s familiarity with the work ———–
SCORE: 3 (Medium (I know the general area/topic))
———– Overall evaluation ———–
SCORE: -1 (Weak Reject (I don’t like the paper, but I could be convinced to accept this paper))

—– TEXT:

This paper proposes a fault-tolerant task scheduling algorithm to improve the overall reliability of the system. To improve the system reliability, the proposed work carefully selects the tasks to be replicated from the task set.

Major comments:

  • Novelty: This paper targets the heterogeneous system of multiple processors to schedule multiple tasks with better fault tolerance. This paper models application by a weighted DAG. Similar DAG based scheduling was introduced by Gao et al. in “AutoPath: Harnessing Parallel Execution Paths for Efficient Resource Allocation in Multi-stage Big Data Frameworks”. As the AutoPath work is also scheduling using DAG model, it would be good to have the paper in discussion and also in the evaluation. Hence, the novelty of the work is not clear.

  • Motivation: From the paper, the motivation of the work is not clear. For example, why exactly fault-tolerance is necessary for scheduling is not well motivated. What are the challenges in finding the task set to be replicated? What are the problems/overhead of selecting incorrect tasks for replication? How can replication in scheduling possibly help to improve the overall performance? Finding answers to these questions will help to understand the problem better. Also, it would be good to have a data-driven motivation that provides the reader a better inspiration for the proposed work.

  • Evaluation: For evaluating the performance of the work, the work is compared with some previous work HEFT and DB-FTSA. However, the paper doesn’t provide a clear picture of the compared work. Specifically, in section V, regarding DB-FTSA, mentions that its scheduling and replication strategy is different from the proposed FTSA-RSMI. But without knowing the details difference, it would be difficult to understand the performance results. Hence, in the analysis, the performance results the proposed work doesn’t explain why it performs better than the other two works. A details in-depth analysis is required to understand the observed similar and better performance using FTSA-RSMI.

  • Results: The result section of the paper includes comparison of the the system reliability varing task numbers, fault rate, execution time, and also for different graph properties. These comparison results are for exploring the reliability improvement. As not all of the results are inspiring, it would be better to have the comparison using other performance metrics like comparison for the scheduling algorithm only. Then it would have both the optimization for the scheduling algorithm and the optimization for the reliability.

—– TEXT:

This paper proposes a fault-tolerant scheduling algorithm with replication strategy of minimum impact of real-time performance (FTSA-RSMI).
The main idea of the proposed FTSA-RSMI algorithm is relatively simple; it continuously adds replication tasks that have the least impact on real-time performance to the task set until either the total execution time of the target task graph reaches the deadline or each task is replicated up to the user-set maximum number.
Evaluation using both synthetically generated applications and real applications (Gaussian elimination, fast Fourier transform, and Molecular dynamics) shows that the proposed FTSA-RSMI algorithm outperforms existing scheduling algorithms (HEFT and DB-FTSA).

[Strength]

  • The main idea and the overall algorithm is intuitive and relatively easy to follow, which may be good to be deployed to real scheduling systems.
  • Evaluation results nicely highlights the benefits of the proposed FTSA-RSMI.

[Weakness]

  • It is unclear of the complexity and scalability of the proposed algorithm; it iteratively finds the next task to replicate using some heuristics, each of which seems to have complexity of at least linear or higher degree complexity.
    However, this paper evaluates the proposed algorithm only in small scale in terms of number of tasks, and there is no information on the incurred overhead for the algorithm calculations.
    Therefore, it will be better if this paper provides complexity analysis of the proposed algorithm either qualitatively or quantitatively so that the feasibility and scalability of the proposed algorithm can be better understood.
  • The proposed algorithm requires the complete knowledge on the execution times of each task on each target processors with their dependencies and incurred communication costs statically, and thus applicability of the proposed scheduling algorithm can be limited.

—– TEXT:

Summary

This paper presents a task replication strategy to improve the reliability of a system while minimizing the impact on real-time performance. The emphasis on heterogeneous systems assuming data dependencies across tasks.

Strengths:

  • The paper is easy to follow

  • Improving the reliability of a distributed system with high transient failure rate is an interesting problem

Weaknesses:

  • The primary weakness of this paper is that it glosses over some important details. For instance, Algorithm 3 which presents the proposed FTSA-RSMI algorithm, Line 17 states “select the scheduling result with the maximum reliability and meeting deadline from the records”. Neither the algorithm nor the corresponding text explains how reliability is computed.

  • The paper makes many simplistic assumptions that would render the proposal impractical. The probability of system failure can be estimated with some degree of accuracy, but the probability of task failure would be approximated to be random. It is not entirely clear how equation (2) could be put to practice.

  • The results are hard to justify given the above.

—– TEXT:

Summary: The paper introduces a fault-tolerant scheduling algorithm with a replication strategy claiming minimum impact on real-time performance called (FTSA-RSMI). The algorithm sets the maximum number of replications for each task. Then, it continuously replicates the tasks that have the least impact on the current system’s real-time performance in the task set to obtain a new task set. The tasks in the new task set will be scheduled and the scheduling results will be recorded. Finally, the scheduling result will be selected as the final scheduling sequence, which has maximum system reliability while meeting the deadline.

  • Pros
    The paper focuses on transient failures and active replication.
    The paper compares to other relevant methods in the literature.
    They use both randomized and real-world applications in the evaluation.
    They achieve better results than techniques they compare to.

  • Cons
    The paper focuses on transient failures only.
    The choice of the techniques to compare against and why this is important could be improved.
    Possibly, some analysis of some of the permanent failures and mitigating them if possible would be interesting.
    The paper could use some editing. It doesn’t hinder from the reading of the paper and understanding but still, it would make much better to have another editorial pass.
